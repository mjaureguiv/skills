# Feature Documentation Template

Use this template for documenting new features or capabilities.

---

## Template Structure

```markdown
# {Feature Title}

{1-2 sentence overview explaining what users can now do with this feature.}

## How {Feature Name} Works

{Explanation of the functionality in user terms. Focus on what happens from the user's perspective, not implementation details.}

### {Subprocess or Component Name}

{If the feature has multiple parts, break them down into subsections. Each subsection should explain one aspect of the feature.}

- {Key point 1}
- {Key point 2}
- {Key point 3}

### {Another Component} (if applicable)

{Continue breaking down complex features into digestible sections.}

## Best Practice: {Recommended Approach}

{If there's a recommended way to use this feature, describe it here. Include specific guidance on configuration or setup.}

{Concrete example with real-world scenario:}

Example: {specific scenario with real values}

{Explain why this approach is recommended.}

## Why This Matters

{Explain the user benefit. What problem does this solve? What can users achieve now that they couldn't before? Focus on outcomes, not features.}

## Getting Started

{Prerequisites or setup steps, if any. Skip this section entirely if the feature is automatically available.}

1. {First step}
2. {Second step}
3. {Third step}

## Limitations

{Any constraints or edge cases users should be aware of. Be factual, not apologetic.}

- {Limitation 1}
- {Limitation 2}

## Related Resources

- [{Related Feature Name}]({link to documentation})
- [{Configuration Guide}]({link})
```

---

## Template Guidelines

### Title
- Use action-oriented language when possible
- State what users can accomplish, not internal feature names
- Good: "Filtering Processes by Lifecycle Status"
- Bad: "SIG-1234 Implementation"

### Overview
- Maximum 2 sentences
- Answer: "What can I do with this?"
- Avoid technical jargon

### How It Works
- Write for the user, not the developer
- Use concrete examples
- Break complex features into subsections

### Best Practice
- Include only if there's a recommended approach
- Show, don't just tell (use examples)
- Explain the "why" behind recommendations

### Why This Matters
- Focus on user outcomes
- Connect to business value
- Avoid marketing language

### Getting Started
- Only include if setup is required
- Keep steps numbered and actionable
- Link to prerequisite documentation

### Related Resources
- Always include at least one link
- Link to parent feature documentation
- Link to configuration guides if applicable
